# Knowledge MCP Server Bug Fixes - 2025-12-08

## Summary
Fixed critical bugs preventing the knowledge-mcp server from loading in Claude Code.

**UPDATE:** After the initial restart, discovered the `.env` file wasn't being loaded. Added `load_dotenv()` call and verified server now starts successfully.

## Issues Found

### 1. Wrong Error Code Constant (22 occurrences)
**Problem:** Server code used `ErrorCodes.UNEXPECTED` but the actual constant in `/srv/latvian_mcp/shared/response.py` is `ErrorCodes.UNEXPECTED_EXCEPTION`.

**Impact:** Would cause AttributeError when any error occurred in the server.

**Files affected:**
- `/srv/latvian_mcp/servers/knowledge-mcp/src/knowledge_mcp/server.py`

**Fix:** Replaced all 22 occurrences of:
```python
ErrorCodes.UNEXPECTED
```
with:
```python
ErrorCodes.UNEXPECTED_EXCEPTION
```

### 2. Missing .env File Loading (Critical)
**Problem:** Server code did not load the `.env` file, so environment variables from `.env` were never read. The `python-dotenv` package was installed but `load_dotenv()` was never called.

**Impact:** Server would fail with "SUPABASE_KEY is required" even though the credentials were in the `.env` file.

**Fix:**
1. Added `python-dotenv` to dependencies in `pyproject.toml`
2. Added import: `from dotenv import load_dotenv`
3. Added `load_dotenv()` call early in server.py (line 32, right after imports)

**Result:** Server now loads `.env` file on startup and reads Supabase credentials correctly.

### 3. Supabase Initialization Timing Bug (Critical)
**Problem:** Supabase client was initialized at module import time (lines 55-65), before Claude Code could inject environment variables from `mcp_servers.json`. This caused the server to fail with "SUPABASE_KEY is required" error during module import.

**Impact:** Server would never start in Claude Code, failing silently. No KB MCP tools would be available.

**Root cause:** The check `if not SUPABASE_KEY: raise ValueError()` executed during `import knowledge_mcp.server`, before the MCP framework could set environment variables.

**Fix:**
1. Changed module-level variables to placeholders:
```python
# Old (at module level):
SUPABASE_URL = os.getenv("SUPABASE_URL", "https://zbhddlduxcwhgibhbeuu.supabase.co")
SUPABASE_KEY = os.getenv("SUPABASE_KEY")
if not SUPABASE_KEY:
    raise ValueError("SUPABASE_KEY is required")
supabase: Client = create_client(SUPABASE_URL, SUPABASE_KEY)

# New (at module level):
SUPABASE_URL: str = ""
SUPABASE_KEY: str = ""
supabase: Client = None  # type: ignore
```

2. Moved initialization to `main()` function:
```python
def main():
    global SUPABASE_URL, SUPABASE_KEY, supabase

    SUPABASE_URL = os.getenv("SUPABASE_URL", "https://zbhddlduxcwhgibhbeuu.supabase.co")
    SUPABASE_KEY = os.getenv("SUPABASE_KEY")

    if not SUPABASE_KEY:
        logger.error("SUPABASE_KEY environment variable not set")
        raise ValueError("SUPABASE_KEY is required")

    supabase = create_client(SUPABASE_URL, SUPABASE_KEY)
    logger.info(f"Connected to Supabase at {SUPABASE_URL}")
    # ... rest of main()
```

**Result:** Module can now import without environment variables, and Supabase client initializes only when `main()` is called with proper env vars set by Claude Code.

## Architecture Notes

### MCP Server Lifecycle:
1. Module is imported by Claude Code
2. Environment variables are injected from `~/.config/claude/mcp_servers.json`
3. `main()` function is called
4. Supabase client is initialized with env vars
5. MCP server starts and registers tools

### Key Insight:
MCP servers must avoid initialization that requires environment variables at module import time. All configuration-dependent initialization should happen in `main()` or be lazy-loaded.
